# 0006 — Role-Based Access Control (RBAC)

## Summary

Add role-based access control to the boilerplate so features can be gated by role (e.g., `admin`, `member`, `viewer`) with a clean server-side enforcement model and a helper for client-side conditional rendering. The database schema is already adapter-compatible; we extend it with a `roles` table and a many-to-many `user_roles` join, expose roles in the JWT session, and enforce them in the edge proxy and server actions.

## Problem / motivation

Today every authenticated user is equal. Real apps need at least `admin` vs `member` gates (admin dashboard, billing, user management). The Auth.js v5 + Drizzle stack is already in place — we just need the data model, session enrichment, and enforcement hooks.

## Goals

- **FR1** — Add `roles` and `user_roles` tables via Drizzle migration (no breaking changes to existing users table).
- **FR2** — Attach `roles: string[]` to the JWT session and `getCurrentSession()` result.
- **FR3** — Edge middleware (`proxy.ts`) can protect route prefixes by role (e.g., `/admin/*` → `admin`).
- **FR4** — Server actions can assert roles via a `requireRole(...)` helper (throws if missing).
- **FR5** — Client components get a `useRole()` hook / `<RequireRole>` wrapper for conditional UI.
- **FR6** — Seed script creates an `admin` role and assigns it to the demo user.
- **FR7** — Settings page displays current user profile (name, email, roles) for all users.
- **FR8** — Admin section in Settings: table of all users with roles, CRUD + role assignment.
- **FR9** — Server actions for admin user management, guarded by `requireRole('admin')`.
- **FR10** — Admin-created users have no password; they set it on first login via `/register` (extended to handle pre-created accounts).
- **NFR1** — Zero Node deps in edge path; role checks in `proxy.ts` stay JWT-only.
- **NFR2** — No breaking changes to existing auth flow; existing users get empty roles array by default.

## Non-goals

- Fine-grained permissions (RBAC only; permissions can be a follow-up spec).
- Organization/team scoping (multi-tenancy is a separate concern).

## Design / approach

### 1. Database schema (`src/db/schema.ts`)

Add two tables:

```ts
export const roles = pgTable('roles', {
  id: text('id')
    .primaryKey()
    .$defaultFn(() => crypto.randomUUID()),
  name: text('name').notNull().unique(), // 'admin', 'member', 'viewer'
  description: text('description'),
  createdAt: timestamp('created_at', { mode: 'date' }).notNull().defaultNow(),
})

export const userRoles = pgTable(
  'user_roles',
  {
    userId: text('user_id')
      .notNull()
      .references(() => users.id, { onDelete: 'cascade' }),
    roleId: text('role_id')
      .notNull()
      .references(() => roles.id, { onDelete: 'cascade' }),
  },
  (t) => [primaryKey({ columns: [t.userId, t.roleId] })],
)
```

Run `pnpm db:generate` → commit migration → `pnpm db:migrate`.

### 2. Seed (`src/db/seed.ts`)

Add `admin` role and assign to `demo@example.com`.

### 3. Session enrichment (`src/lib/auth/session.ts`)

Modify `getCurrentSession()` to attach `roles: string[]` (names, not IDs) by:

- Reading `user.roles` from the JWT (populated at sign-in).
- If missing (old tokens), fall back to a one-time DB lookup and refresh the JWT via `update()`.

### 4. JWT population (`src/lib/auth/index.ts`)

In the Credentials `authorize()` callback, after verifying password:

- Query `userRoles` + `roles` for that user.
- Add `roles: string[]` to the returned user object → ends up in JWT.

### 5. Edge route protection (`src/proxy.ts`)

Add a `roleRequired` map (or matcher function) alongside the existing `protectedPrefixes`:

```ts
const roleRequired: Record<string, string[]> = {
  '/admin': ['admin'],
  '/billing': ['admin', 'member'],
}
```

Check `token.roles?.includes(required)` before allowing.

### 6. Server action guard (`src/lib/auth/rbac.ts`)

```ts
export function requireRole(...allowed: string[]): void {
  const session = await getCurrentSession()
  if (!session?.user.roles?.some((r) => allowed.includes(r))) {
    throw new ForbiddenError('Insufficient role')
  }
}
```

Use in server actions: `requireRole('admin')`.

### 7. Client helpers (`src/lib/auth/client-rbac.tsx`)

- `useRole()` → returns `string[]` from `useSession()`.
- `<RequireRole roles={['admin']} fallback={<AccessDenied />}>...</RequireRole>`.

### 8. Types (`src/lib/auth/types.ts` or extend `config.ts`)

Extend `User` and `JWT` interfaces to include `roles: string[]`.

### 9. Admin server actions (`src/lib/auth/admin-actions.ts`)

Server actions for user management, all guarded by `requireRole('admin')`:

```ts
// Create user (no password) — admin invites someone
export async function createUserAction(data: { name: string; email: string }) { ... }

// Update user profile (name, email) — admin edits
export async function updateUserAction(userId: string, data: { name?: string; email?: string }) { ... }

// Delete user — admin removes (cascades to user_roles)
export async function deleteUserAction(userId: string) { ... }

// Assign/replace roles for a user
export async function setUserRolesAction(userId: string, roleIds: string[]) { ... }
```

### 10. Settings page — current user + admin panel (`src/app/(dashboard)/settings/page.tsx`)

- **Current User Card** (all signed-in users): Shows name, email, roles badges.
- **Admin Panel** (`RequireRole roles={['admin']}`):
  - Table: all users with email, name, roles (multi-select), actions (edit/delete).
  - "Add User" button → modal form (name, email) → creates user without password.
  - Edit row → modal (name, email).
  - Delete row → confirmation.
  - Role changes applied instantly via `setUserRolesAction`.

### 11. Extended register flow (`src/lib/auth/actions.ts`)

The `registerAction` already supports creating an account. Extend it to handle **pre-created users** (admin-created, no `hashedPassword`):

- If user exists with that email AND `hashedPassword` is null → treat as "claiming" the account.
- Validate password, hash, update user, sign in.
- If user exists with password → error "account exists".

This lets admins create users who then set their own password on first visit.

### Deferred hardening (follow-ups)

- Invite tokens / email verification for account-claim (currently anyone knowing
  a pre-created email can claim it).
- Associate `FormLabel` with its input (a11y — inputs are currently announced by
  placeholder).
- Focus-trap/Escape on the hand-rolled delete modal (use the shadcn `Dialog`).
- Consolidate the two `next-auth` type-augmentation files.

## Security & privacy

- Roles live only in DB + JWT (no localStorage).
- Edge proxy checks JWT claim — no DB round-trip on protected routes.
- Server actions re-validate via `getCurrentSession()` (DB-backed if token stale).
- No role escalation path exposed to client.

## Alternatives considered

- **NextAuth `role` single string** — too limiting; many apps need multiple roles.
- **CASL / external lib** — overkill for boilerplate; native JWT + helpers is lighter.
- **Postgres enum for roles** — flexible `text` + unique index is simpler for migrations.

## Out of scope / future

- Permission strings (`posts:create`, `billing:read`) — can layer on top later.
- Admin UI for role assignment.
- Multi-tenant / organization-scoped roles.

## OAuth Provider Integration

The schema is already Auth.js Drizzle adapter compatible. When OAuth providers (GitHub, Google, etc.) are added:

### Adapter Setup (in `src/lib/auth/index.ts`)

```ts
import { DrizzleAdapter } from '@auth/drizzle-adapter'

export const { handlers, auth, signIn, signOut } = NextAuth({
  ...authConfig,
  adapter: DrizzleAdapter(db),
  providers: [
    Credentials(...),
    GitHub({ clientId, clientSecret }),
    Google({ clientId, clientSecret }),
  ],
  callbacks: {
    ...authConfig.callbacks,
    async signIn({ user, account, isNewUser }) {
      if (isNewUser && account?.provider !== 'credentials') {
        const memberRole = await db.query.roles.findFirst({
          where: eq(roles.name, 'member'),
        })
        if (memberRole) {
          await db.insert(userRoles).values({
            userId: user.id,
            roleId: memberRole.id,
          }).onConflictDoNothing()
        }
      }
      return true
    },
  },
})
```

### Session Enrichment (works for ALL providers)

The `jwt` callback in `authConfig` populates `token.roles` for both Credentials and OAuth users:

```ts
jwt: async ({ token, trigger }) => {
  if (trigger === 'update' || !token.roles) {
    token.roles = await getUserRoles(token.id) // cached DB lookup
  }
  return token
}
```

### Role Linking Across Providers

- Auth.js adapter links accounts by email (case-insensitive unique index exists in `users` schema)
- Same `user.id` → same `user_roles` rows → consistent roles regardless of provider
- Works automatically once adapter is enabled

### Bootstrap Options

| Strategy           | Implementation                                           |
| ------------------ | -------------------------------------------------------- |
| First user = admin | `signIn` callback: `count(users) === 0` → assign `admin` |
| Invite codes       | Separate `invites` table with `role` claim               |
| Manual DB          | `INSERT INTO user_roles ...` (simplest for bootstrap)    |

**Default for this spec**: First user (any provider) gets `admin`; subsequent OAuth users get `member`.
