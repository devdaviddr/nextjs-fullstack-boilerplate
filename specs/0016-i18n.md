# 0016 — Internationalization (i18n)

## Summary

Add locale-prefixed routing and message-based translation via `next-intl`,
proven end-to-end with English plus one additional locale. This is the
largest, most broadly-touching spec in the roadmap — it restructures every
route under `app/[locale]/...` and requires extracting every user-facing
string — and is scoped and sequenced accordingly.

## Problem / motivation

English-only is a perfectly valid default for a personal portfolio
boilerplate, and this spec is explicitly the lowest-priority, most-deferrable
item on the roadmap for exactly that reason. It's still worth having a
concrete plan for the fork that does need it, rather than leaving it as an
unscoped "someday."

## Goals

- Locale-prefixed routes (`/en/...`, `/es/...`) with correct locale
  detection/redirect.
- All existing user-facing UI strings extracted to message files, proven with
  two locales end-to-end (not just infrastructure with untranslated stubs).
- A documented, mechanical path for a fork to add locale #3 without touching
  routing/middleware code.

## Non-goals

- Translating email templates (`src/lib/email/templates.ts`) — flagged as a
  deliberate follow-up to bound this spec's size; email is already
  opt-in/off-by-default, and localizing it doubles the string-extraction
  surface for a feature that's already the largest single change here.
- Right-to-left (RTL) layout support — no RTL locale is in the initial scope;
  documented as a follow-up if a fork needs one.
- Automated/AI-assisted translation — message files are hand-authored or
  fork-owner-supplied for the initial two locales.

## Requirements

### Functional

- **FR1** — `next-intl` added; `src/app/` restructured so every existing
  route (`(auth)`, `(dashboard)`, `403`, `offline`, etc.) moves under
  `app/[locale]/...`, preserving the existing route groups.
- **FR2** — `messages/en.json` and `messages/es.json` (or another second
  locale — chosen for maximal test coverage of pluralization/interpolation
  differences from English) with every current UI string extracted:
  auth forms, admin panel, error/403/offline pages, the app shell nav.
- **FR3** — `proxy.ts` composes `next-intl`'s middleware helper for locale
  detection/redirect **alongside** the existing auth/RBAC/CSP logic already
  in that file — ordering matters (locale resolution before auth checks; CSP
  nonce applies unconditionally regardless of locale) and is explicitly
  designed, not incidental.
- **FR4** — A locale switcher in the app shell and on `/login`/`/register`.
- **FR5** — `docs/i18n.md`: the exact, mechanical steps to add a new locale
  (add a message file, add the locale to the supported-locales list) —
  written so a fork owner never needs to touch routing code.

### Non-functional

- **NFR1** — Compatible with Turbopack (both `dev` and `build`) — this repo
  is Turbopack-only per `CLAUDE.md`; `next-intl` is chosen specifically
  because it's App-Router-native and doesn't require the webpack-oriented
  tooling older i18n libraries (e.g. `next-i18next`) depend on.
- **NFR2** — No regression to existing route protection, RBAC, or CSP nonce
  behavior after `proxy.ts` gains locale-handling — the existing test suite
  (unit + E2E) continues to pass unmodified in structure, only path prefixes
  change.
- **NFR3** — SEO metadata ([0008](0008-seo-metadata.md), if shipped first)
  gains per-locale `hreflang` alternates — sequencing note, not a hard
  dependency (this spec doesn't require 0008, but should account for it if
  both exist).

## Design / approach

- Route restructuring is mechanical but large: every `page.tsx`/`layout.tsx`
  under `src/app/` moves one level deeper under `[locale]`. Do this as a
  dedicated, reviewable commit before any string extraction, so the
  file-move diff is separable from the translation diff.
- `proxy.ts` ordering: resolve/redirect locale first (so `pathname` checks
  for `PROTECTED_PREFIXES`/`AUTH_ROUTES`/`ROLE_REQUIRED` operate on the
  locale-stripped path), then run the existing auth/RBAC logic unchanged,
  then apply the CSP nonce unconditionally at the end — the nonce/CSP logic
  has no locale dependency and should not be duplicated per-locale.
- Message extraction starts from the auth flow and admin panel (the largest
  concentration of strings) and works outward; email templates are
  explicitly deferred (Non-goals).
- Locale detection: `Accept-Language` header on first visit, then a cookie
  once a locale is resolved/chosen — standard `next-intl` behavior, not
  custom logic.

## Acceptance criteria

- [ ] Visiting `/` with no locale redirects to the detected/default locale
      prefix.
- [ ] Every string on the auth pages, admin panel, error/403/offline pages,
      and app shell nav renders from a message file in both shipped locales
      — no hardcoded English strings remain outside `en.json` (grep-verified).
- [ ] Switching locale via the switcher updates the UI and the URL prefix
      without a full page reload where avoidable.
- [ ] `proxy.ts` route protection, RBAC gating, and CSP nonce all continue to
      work identically per-locale (existing E2E suite passes against
      locale-prefixed paths).
- [ ] `docs/i18n.md`'s "add a locale" steps are followed literally to add a
      third, throwaway locale during review, with zero routing-code changes.
- [ ] `pnpm lint && pnpm typecheck && pnpm test && pnpm test:e2e && pnpm build`
      pass.

## Security & privacy

- No new attack surface beyond standard `next-intl` locale-cookie handling;
  the locale cookie is not security-sensitive (unlike the session cookie).
- CSP nonce logic in `proxy.ts` must not be weakened or duplicated
  incorrectly during the middleware composition (NFR2/FR3) — this is the one
  place a careless merge could regress an existing security control.

## Alternatives considered

- **`next-i18next`** — the long-standing standard, but built around the
  Pages Router and webpack-era conventions; doesn't fit this repo's App
  Router + Turbopack-only constraints without extra shims.
- **Roll a custom message-lookup solution (no library)** — avoids a
  dependency, but reimplements plural rules, interpolation, and locale
  negotiation that `next-intl` already gets right; not worth it for a
  boilerplate meant to demonstrate correct patterns.
- **Cookie-based locale with no URL prefix** — simpler routing, but hurts
  SEO (search engines can't discover/index per-locale content) and makes
  sharing a locale-specific link impossible; rejected given
  [0008](0008-seo-metadata.md)'s goals.

## Out of scope / future

- Email template localization.
- RTL layout support.
- AI-assisted or crowd-sourced translation workflows.
