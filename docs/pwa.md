# PWA & App Shell

[← Back to README](../README.md)

The app is installable and offline-resilient with **no extra runtime dependencies and no bundler change** — it stays entirely on Turbopack. The service worker is hand-rolled rather than generated by a Webpack-only plugin.

## What makes it a PWA

| Piece                                               | File                                             |
| --------------------------------------------------- | ------------------------------------------------ |
| Web app manifest → `/manifest.webmanifest`          | `src/app/manifest.ts`                            |
| Icon set (192, 512, maskable, apple-touch, favicon) | `scripts/generate-icons.mjs`                     |
| Service worker                                      | `public/sw.js`                                   |
| SW registration + update flow                       | `src/components/pwa/service-worker-register.tsx` |
| Install prompt (`beforeinstallprompt`)              | `src/components/pwa/install-prompt.tsx`          |
| Offline fallback page                               | `src/app/offline/page.tsx`                       |
| Theme color · iOS · safe-area viewport              | `src/app/layout.tsx`                             |
| `/sw.js` no-cache header                            | `next.config.ts`                                 |

## Caching strategy

The service worker's caching is deliberately strict because this is an **authenticated** app — caching personalized responses in a shared cache would leak one user's data to the next person on the device.

| Request                                 | Strategy                                                       |
| --------------------------------------- | -------------------------------------------------------------- |
| `/_next/static/*`, icons, fonts, images | **cache-first** (immutable, fingerprinted)                     |
| Page navigations                        | **network-only**, falling back to `/offline` when disconnected |
| `/api/*`, `/api/auth/*`                 | **never cached** — always network                              |
| Non-GET requests                        | passed straight through                                        |

To invalidate all caches on the next deploy, bump `CACHE_VERSION` in `public/sw.js`.

## Registration & updates

- The SW registers **in production only** — dev-mode service workers cause confusing caching during development.
- Registered with `updateViaCache: 'none'` and served with `Cache-Control: no-cache`, so clients always revalidate `/sw.js` and pick up new versions.
- When a new worker is waiting, it is activated and the page reloads once.

To test the installable app locally:

```bash
pnpm build && pnpm start
# then: Chrome DevTools → Application → Manifest / Service Workers,
# or run a Lighthouse "Installable" audit.
```

## Icons

Placeholder icons are generated from a single inline SVG:

```bash
pnpm gen:icons
```

To rebrand, edit the SVG mark and colors in `scripts/generate-icons.mjs` (or drop in real source art) and re-run. Update the name/colors in `src/app/manifest.ts` to match.

## App shell

Protected routes render inside a minimal, borderless shell (`src/components/shell/`):

- **Desktop:** fixed sidebar + sticky topbar.
- **Mobile:** sidebar collapses to an off-canvas drawer (hamburger toggles it; closes on navigation and backdrop tap).
- **Installed PWA:** `env(safe-area-inset-*)` padding keeps the topbar and drawer clear of the notch.
- **Navigation** is data-driven from `src/lib/shell/nav.ts` — add `{ title, href, icon }` and it appears with active-state highlighting.

Public pages (`/`, `/login`, `/register`) keep their own centered layout.

## Push notifications

Web Push is wired up (opt-in). The service worker handles `push` (shows the
notification) and `notificationclick` (focuses or opens the right tab);
subscriptions are stored per device in `push_subscriptions`, managed from a
Settings toggle, and sent server-side via `src/lib/push/`. It stays fully inert
until the `VAPID_*` env vars are set. See
[Features → Web Push](features.md#web-push-notifications-optional) and
[spec 0015](../specs/0015-web-push-notifications.md).
